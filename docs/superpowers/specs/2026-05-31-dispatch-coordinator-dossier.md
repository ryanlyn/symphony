# Ground-Truth Dossier: Dispatch/Pool Coordinator (MachineLease vs RunSlot, N-agents-per-machine)

## 1. The Core Conflation You Are Splitting

Today a single `BoxRecord` (in `ts/packages/worker-box-pool/src/types.ts`) and a single set of operations on `BoxPoolImpl` (`ts/packages/worker-box-pool/src/pool.ts`) carry **two distinct lifecycles fused into one object**:

- **Machine identity / lifecycle** (long-lived): `boxId`, `workerHost`, `providerRef`, `createdAtMs`, `labels`, `metadata`, `state` (`WARM_IDLE` / `LEASED` / `DRAINING` / `DESTROYED`), `boxSecondsUsed`, `expiresAtMs = createdAtMs + ttlMs`. Driven by provision/warm/idle/ttl/drain/reaper/ledger/spend.
- **Run lease state** (per-run, stamped onto the same record): `leaseId`, `inFlight`, `leaseIssues` (a `Set`), `liveLeaseAcquiredMs` (per-window billing stack), `affinityKey`, `markedForDestroy`.

The conflation is concretely visible in `stamp` (pool.ts:601-632): acquiring a run mutates the machine record in place — `record.leaseId = randomUUID()`, `record.inFlight += 1`, `record.state = "LEASED"`, `record.leaseIssues.add(req.issueId)`, `record.affinityKey = record.workerHost`, `record.liveLeaseAcquiredMs.push(...)`. The same `Map<boxId, BoxRecord>` (pool.ts:89), the same per-box mutex (pool.ts:93), and the same `state` field encode both "this machine exists and is warm" and "this run currently occupies the machine." `maxInFlight>1` already lets one machine carry N concurrent leases (`isLeasable` pool.ts:56-60; under-capacity branch in `pickRecord` pool.ts:588-597; `stamp` increments `inFlight` without a fresh machine), and `onLeaseSettle` only returns the machine to `WARM_IDLE` (or recycles it) at `inFlight==0` (pool.ts:914-929).

**This is the seam for your MachineLease vs RunSlot split.** A clean model:
- **MachineLease** ≈ what's left of `BoxRecord` once run state is removed: identity, provider ref, state, ttl, spend, mutex. Owned by the pool, reaped/drained/ledgered as today.
- **RunSlot** ≈ a first-class per-(issueId, slotIndex, leaseId) object that *references* a MachineLease and owns the per-run sub-resources the machine does NOT currently isolate (the MCP tunnel, the forwarded port). Today a "run" has no identity beyond the `leaseId` stamped on the machine; that is exactly what must become first-class.

The reason this split has never been forced is that **`maxInFlight` defaults to 1**, so machine == run and the fusion is harmless. The hardening already present for `maxInFlight>1` (overlapping `liveLeaseAcquiredMs` windows + per-box mutex serializing release-vs-reaper) is **purely about box-second billing and inventory consistency** — it does NOT make the in-machine runtime resources safe to share. That is the blocker below.

## 2. THE BLOCKER: MCP-Tunnel-vs-Recycle Coordination Gap (why N-per-machine is unsafe today)

This is the single load-bearing reason `maxInFlight=1` is the only safe value, and the central problem your coordinator must solve.

**Two pools refcount independently and never coordinate:**
- **Box pool**, keyed on `boxId` / `workerHost`, accounts capacity via `inFlight` (`onLeaseSettle` pool.ts:868-934) and tears down machines via `recycle` (pool.ts:969-999) and `runDrain` (pool.ts:1121-1190).
- **Tunnel pool** (`WorkerHostPool`, `ts/packages/worker-host-pool/src/index.ts`), a **process-global singleton** (`export const workerHostPool` index.ts:103), keyed **solely on `workerHost`**: `remoteMcpTunnels = Map<workerHost, RemoteMcpTunnelEntry>` (index.ts:19). `acquireRemoteMcpTunnel(workerHost, localHost, localPort)` refcounts a shared entry if one exists with matching `localHost/localPort` (`refCount += 1`, returns the **same** `remotePort`, index.ts:26-35); otherwise allocates one `remotePort` per host (`nextRemoteMcpPort` / `availableRemoteMcpPorts`, starting 46000, index.ts:38-47) and starts ONE `ssh -N -R remotePort:localHost:localPort` reverse tunnel. `releaseRemoteMcpTunnel(workerHost)` decrements and only kills the ssh child at the last ref (index.ts:73-81).

**The path that makes it load-bearing:** `acp/src/index.ts:81` calls `acquireAgentMcpEndpoint(settings, workerHost)` → `mcp/src/agentEndpoint.ts:96-112` calls `workerHostPool.acquireRemoteMcpTunnel(workerHost, '127.0.0.1', localPort)` and hands the agent the URL `http://127.0.0.1:${tunnel.remotePort}/claude-mcp`. For a remote run, `clientCapabilities(workerHost)` drops `fs.readTextFile/writeTextFile` (acp index.ts:517-526), so **the MCP tunnel is the ONLY `symphony_linear` channel back to the controller** — losing it kills the run's tool access.

**Three concrete race modes when N>1 leases share one `workerHost`:**
1. **Recycle out from under a sibling.** Run A poisons the box → `lease.fail('box_poisoned')` → `recycle` destroys the MACHINE based purely on the box pool's own `inFlight==0` accounting, with **NO call into `releaseRemoteMcpTunnel`**. Run B, still LEASED on that host, keeps believing its `workerHost`-keyed tunnel URL is valid. The orphaned `ssh` child keeps running or errors out; the box is gone.
2. **Last-ref kill mid-flight.** The first run to finish calls `releaseRemoteMcpTunnel(workerHost)`; if it were the last ref it kills the shared tunnel out from under a still-running sibling.
3. **localPort swap on reload.** Each acquire passes `localPort = localServer.handle.port` (agentEndpoint.ts:101-102). If a config reload swaps that port, the SECOND acquire's mismatched `localHost/localPort` triggers `closeRemoteMcpTunnel` of the **live shared tunnel** (index.ts:30-36), tearing down the first run's MCP mid-flight.
4. **Port collision after recycle.** A recycled `remotePort` can be re-minted for a new box while a stale tunnel entry the `WorkerHostPool` still has cached as not-exited lingers.

The box pool's generation guards (`leaseId` + `settled` + box-state, per-box mutex) **do not extend to the tunnel**. The domain comment (`domain/src/index.ts:191`) names 1 "the only value safe against the workerHost-keyed MCP tunnel refCount." Critically, **`maxInFlight` is NOT clamped to 1 in config** — `config.ts:594` is `positiveInt(...,1)` (default 1, but any value passes), and `isLeasable`/`pickRecord`/`stamp` all honor N. **The safety is by-convention/default only.** A naive bump to 2 is silently corrupting.

**Design implication for the coordinator:** the RunSlot abstraction must own a **per-run (per-lease/per-slot) tunnel + forwarded port**, acquired and released on the **exact same exactly-once boundary as the lease** (see §4), with generation-safety so a late release cannot kill a newer run's tunnel. Either re-key the tunnel by `(workerHost, leaseId/slotIndex)` instead of `workerHost` alone, or introduce a lease→tunnel refcount handshake so box recycle/drain cannot proceed while a tunnel ref is outstanding (and cannot orphan one). Today neither exists.

## 3. What Is Already Per-Run-Safe (do NOT re-solve, do NOT regress)

On a shared machine, **filesystem state is already isolated per issue/slot**; only **network/tunnel/port state is per-host**. The split is clean:

- **Remote workspace dirs** — `remoteWorkspacePath(root, identifier, slotIndex, ensembleSize)` (`workspace/src/index.ts:366-374`) → `<root>/<safeIdentifier(identifier)>`, or `<root>/<identifier>/<slotIndex>` when `ensembleSize>1`. Two issues, or two ensemble slots, on the same box get distinct dirs (`createRemoteWorkspaceForIssue` index.ts:256-301, keyed on workerHost+identifier+slot). `ensureRemoteInsideRoot`/`validateWorkspaceCwd` box each run under root.
- **Resume-state keying** — `resumeStatePath` → `<gitDir>/symphony/resume.json` **inside the per-slot workspace** (`resume-state/src/index.ts:91-121`), so it is implicitly isolated by the workspace dir. `resumeStateMatches` additionally pins `{agentKind, issue.id, issue.identifier, issue.state, workspacePath, workerHost}` (resume-state index.ts:143-156) and `writeResumeState` carries `workerHost` — a mismatched host/issue refuses resume.

**One sharp edge to note:** when `ensembleSize==1`, both `workspacePath` and `remoteWorkspacePath` return the bare `<root>/<identifier>` with **no slot suffix** (workspace index.ts:22, 373). Safe today (one issue == one slot), but **two leases for the same issue on one box** (a `maxBoxesPerIssue` / `maxInFlight` interaction) would share the workspace + resume path and clobber. If your coordinator ever lets two slots of the same issue co-reside, you must force a slot suffix unconditionally.

**Hard constraint:** the pool must NOT touch workspaces or hooks. `createBoxPool` takes only `{clock, logEvent, ledgerPath}` (pool.ts:33-43); the runner owns workspaces/resume-state. No workspace/hook deps may enter `@symphony/worker-box-pool`.

## 4. The Orchestrator↔Runtime Seam (the contract any coordinator must preserve)

Three pieces: **CapacityProbe**, the **`pending://` sentinel two-phase host bind**, and **acquire-before-runner / release-in-finally**.

**CapacityProbe** (`orchestrator/src/index.ts:57-60`): `{ governs(): boolean; canAcquire(): boolean }`, injected as the 4th Orchestrator ctor arg (index.ts:69). Constructed in `runtime/src/index.ts:300-302` as `boxPool ? { governs: () => boxPool.isEnabled(), canAcquire: () => boxPool.canAcquire() } : undefined`.
- `governs()` mirrors `settings.enabled` (`pool.isEnabled` pool.ts:269). When true, `claim()` and `workerCapacityAvailable()` delegate to the pool and bypass static `sshHosts` selection.
- `canAcquire()` (pool.ts:251-261): false if disabled/draining/spend-exhausted, true iff a leasable box or growth headroom exists. Consumed by `workerCapacityAvailable()` (orchestrator index.ts:214-222), which feeds `dispatchState.workerCapacityAvailable` into `eligibleIssues` (index.ts:90) and `claim` (index.ts:120).

**Two-phase host bind (the `pending://` sentinel):**
- **Phase 1 — synchronous, in the poll thread.** `Orchestrator.claim` (index.ts:107-177): when `capacityProbe?.governs()` (index.ts:134), sets `entry.workerHost = pending://${issue.id}/${slotIndex}` (index.ts:137) and `entry.affinityHost = retry?.workerHost ?? null` (index.ts:138). The `RunningEntry` enters `state.running`/`state.claimed` **synchronously** carrying the sentinel (index.ts:173-174). When NOT governing, falls back to static `selectWorkerHost()` (index.ts:139-145, 201-212).
- **Phase 2 — async, in `runClaim`.** After `boxPool.acquire(...)` returns a lease, `effectiveWorkerHost = lease.workerHost` then `orchestrator.setWorkerHost(issue.id, slotIndex, effectiveWorkerHost)` (runtime index.ts:536-540) overwrites the sentinel (orchestrator index.ts:183-187; no-op if the slot is gone).

**The window** is `[claim() returns ... setWorkerHost fires]`, i.e. while `acquire()` is in flight (can block up to `acquireTimeoutMs` on the FIFO waiter queue). **Every reader of `RunningEntry.workerHost` in that window must stay sentinel-tolerant:**
- `cleanupWorkerHost` (runtime index.ts:1054-1056) maps any `pending://`-prefixed host → `null` so reconcile-driven workspace/resume cleanup stays **LOCAL** instead of SSHing to the bogus host. Used in `reconcileTrackedIssues` (index.ts:698, 703-705).
- `snapshot()` → `runtimeRunningEntry` copies `entry.workerHost` straight through (runtime index.ts:984-1007, field at 997) → surfaces the literal `pending://...` to TUI/web/history. **Intentionally never null** so it does not misleadingly read as "local."
- `finish` copies `entry.workerHost` into `RetryEntry.workerHost` (orchestrator index.ts:274-275) → a retry could carry the sentinel as next-claim `affinityHost`; harmless (affinity is a best-effort hint).
- History rows (`entry?.workerHost`, runtime index.ts:583, 623) normally read post-`setWorkerHost`. The one exception: `reconcileStalledRuns` (index.ts:792) could record the sentinel if a stall fires before acquire resolves — but acquire shares `handle.signal`, so a stall abort cancels the acquire first.
- `selectWorkerHost` least-loaded counter (orchestrator index.ts:201-212) counts `entry.workerHost`, but only on the **non-governing** static path where the sentinel never appears — benign.

**Acquire-before-runner / release-in-finally** (`runtime/src/index.ts:483-648`), guarded by `if (this.input.boxPool && this.input.boxPool.isEnabled())` (501):
- **Acquire** at 504-511: `boxPool.acquire({ issueId, slotIndex, labels, affinityKey: affinityHost, timeoutMs, signal: handle.signal })`.
- **Acquire rejection** (thrown — ledger/fs/provider fault) 512-526: emit `box_pool_acquire_error`, `abandonClaim` + `handle.release()`, return — **no run, no history, no lease to settle**.
- **No-capacity** (`status !== "leased"`) 527-535: emit `worker_host_capacity` `dispatch_skipped`, `abandonClaim` + `handle.release()`, return — **no retry/backoff**.
- **Lease bind** 536-540: `lease = acquired.lease; effectiveWorkerHost = lease.workerHost; setWorkerHost(...)` before the runner (544).
- **Heartbeat** 542/550: `heartbeatLease?.heartbeat()` on every agent `onUpdate`.
- **The finally** 635-647: `handle.release()`, then if `lease`: `boxOutcome === "poison" ? await lease.fail("box_poisoned") : await lease.release("healthy")`. `boxOutcome` is classified at 599 (`handle.reason === "stalled"` → poison, else `classifyBoxOutcome(error)`).

`abandonClaim` (orchestrator index.ts:195-199) drops running+claimed with **no retry/backoff** so the slot is immediately re-claimable — the inverse of `claim`, distinct from `finish` (index.ts:246-279) which schedules backoff.

- `pool.ts:209-249` — `acquire()`: gates enabled/draining/spend, then (1) synchronous `selectAndStamp`, (2) `grow` under reservation, (3) FIFO `waitForCapacity`. Returns `{status:'leased',lease}` or `{status:'no_capacity',reason}`.

- `pool.ts:797-857` — `waitForCapacity`/`settleWaiter`/`wakeWaiters`: FIFO waiter queue; `wakeWaiters` re-runs `selectAndStamp` for the oldest waiter on any capacity-freeing event, or fails waiters `pool_disabled` while draining.

- `pool.ts:868-934` — `onLeaseSettle()`: rolls day key, `inFlight--`, `leaseIssues.delete`, drops the `liveLeaseAcquiredMs` window, accrues box-seconds, and at `inFlight==0` either recycles (poison || `markedForDestroy`), leaves `WARM_IDLE` for drain, or returns `WARM_IDLE` healthy. Wakes drain barrier + waiters.

- `pool.ts:1121-1190` — `runDrain()`: stop reaper, reject waiters, await `inFlight→0` to deadline (epoch-guarded), `accrueInFlightWindows` + recycle every machine; `drainEpoch` bail so a re-enabled pool's fresh boxes survive.
- Runtime seam: `runtime/src/index.ts:504-540` (acquire+setWorkerHost) & `635-647` (settle once). Acquire-failed → `abandonClaim` (orchestrator index.ts:195-199).

## 7. FINDING #1 — provider-change not rebuilt on reload (a known gap to fix carefully)

`this.provider` is resolved **once** in the constructor — `resolveProvider(settings.provider, settings, {...})` (pool.ts:161-164) — and stored as `private readonly provider: BoxProvider` (137). `reconcile()` assigns `this.settings = next` and `this.reaperInternals.settings = next` but **never re-runs `resolveProvider` and never reassigns `this.provider` (it is readonly)**. So a hot-reload that changes `worker.box_pool.provider` (e.g. static-ssh → docker/fly) or any provider-construction input baked in at factory time (static-ssh's `ssh_hosts`, a cloud image/app) leaves the **live pool provisioning/probing/destroying/listing against the OLD provider** while reporting the NEW settings everywhere (`snapshot().provider` pool.ts:541 reflects `next`). Affected sites running the stale provider: `acquire→grow→provider.provision` (pool.ts:728), `provisionWarm` (1066), `recycle→provider.destroy` (973), `reconcileWithProviderList→provider.list` (reaper.ts:129), `reapUnhealthy→provider.probe` (276). New ledger/spend/reservation accounting uses the new settings, so **the books and the actual backend diverge**. The daemon does not paper over it: `buildBoxPool` (`apps/cli/src/daemon.ts:55-61`) constructs once at startup; reload flows exclusively through `reconcile`. Effectively the provider is pinned for the singleton's lifetime — only a process restart picks up a change.

**The constraint your fix must honor:** the pool is a **reload-surviving singleton and must never be reconstructed**. So fixing #1 requires `reconcile` to **rebuild/swap the provider in place** (re-run `resolveProvider`, re-thread the new provider into `reaperInternals` + `ledger`/`usesLedger`, and presumably drain old-provider boxes), NOT reconstruct `BoxPoolImpl`. Treat #1 as either explicitly fixed this way or explicitly preserved — never silently changed.

## 8. Invariants to Preserve (the non-negotiable checklist)

1. **Orchestrator stays PURE + SYNCHRONOUS.** `claim`/`setWorkerHost`/`abandonClaim`/`finish` (orchestrator index.ts:107-279) do no I/O, no awaits, mutate only in-memory maps. The **CapacityProbe `{governs,canAcquire}` is the ONLY pool seam** and is read synchronously. All async acquire/release lives in `runtime.runClaim`. The host-binding decision in `claim()` must stay synchronous (the `pending://` sentinel); the real address resolves only later in `runClaim`.

A stale generation's late resolve must remain a no-op that never touches `inFlight`. Generation-scoped active-run cleanup (commit 8a8b89a) must hold. **Tunnel teardown must be generation-safe the same way** — a late release must not kill a newer run's tunnel.
4. **Hardened machine internals unchanged.** Synchronous `reservedProvisions` + `reservedProvisionsByIssue` single-flight growth (can't exceed `max` / `maxBoxesPerIssue`); per-box mutex; recurring reaper with `unref` + in-progress guard; `drainEpoch` barrier + re-enable invalidation; write-ahead ledger (provisional→correlate) + `spend.json` daily sidecar with UTC roll; box-seconds accounting incl. `accrueInFlightWindows` on drain. 5. **Drain never leaks a paid cloud box.** `runDrain` force-destroys every machine after the deadline, AND `grow`/`provisionWarm` destroy a machine that lands mid-drain (pool.ts:750-753, 1081-1084). `hydrate` re-adopt + `provider.list()` reconcile stays authoritative (never wipe inventory on a transient `list()` failure).
6. **`maxInFlight=1` invariant respected until the tunnel-vs-recycle gap is closed.** This is the gating safety constraint. Raising it requires the §2 coordination to exist. The single-tenant path must stay **byte-for-byte** unchanged.
7. **`pending://` sentinel contract.** `claim` sets a **non-null** `pending://<id>/<slot>` (never null, to avoid a misleading "local" projection); `setWorkerHost` overwrites it before the runner; `cleanupWorkerHost` maps it back to `null` so reconcile cleanup stays local. Every `workerHost` reader in the window must keep tolerating the sentinel; any new per-run host identity must not re-introduce SSH to a sentinel host.
8. **Per-run isolation that works must not regress.** Per-issue/slot remote workspace dirs (`remoteWorkspacePath`) and workspace-rooted resume-state keying. The pool must NOT touch workspaces or hooks (no workspace/hook deps in `@symphony/worker-box-pool`; `createBoxPool` takes only `{clock, logEvent, ledgerPath}`).

10. **Absent-pool path stays byte-for-byte.** No `boxPool` → orchestrator takes static `sshHosts`/local selection, `runClaim` skips acquire entirely (probe only installed when a pool exists).

## 9. Synthesis: What the Coordinator Design Must Do

- **Make RunSlot first-class.** Lift run-lease state (`leaseId`, `inFlight` participation, `leaseIssues` membership, `liveLeaseAcquiredMs` window, `affinityKey`) off `BoxRecord`/MachineLease into a per-(issueId, slotIndex, leaseId) object. Keep `affinityKey = workerHost` semantics (it keys per-machine for sticky reuse, pool.ts:606/928). The `stamp`/`onLeaseSettle` pair is the extraction site.
- **Close the tunnel-vs-recycle gap before allowing `maxInFlight>1`.** Give each RunSlot its own MCP tunnel + forwarded remote port (re-key `WorkerHostPool` by `(workerHost, slot/leaseId)` rather than `workerHost` alone, or add a lease→tunnel refcount handshake) so that: recycle/drain of a MachineLease cannot proceed while any RunSlot holds a tunnel ref; tunnel acquire/release rides the exactly-once lease boundary; and a poisoned run cannot orphan or steal a sibling's tunnel. This is the gating work — everything else (workspace, resume) is already per-slot safe.
- **Force a slot suffix** on workspace/resume paths if two slots of one issue may co-reside (the `ensembleSize==1` bare-path edge).
